# Troubleshooting Clan Commands

This document describes common issues and debugging steps for clan CLI commands.

## Background: How Clan Manages Machine Keys

Clan uses a **generated age keypair** for each machine (NOT SSH-derived keys by default):

| Component                   | Location                                | Description                                 |
| --------------------------- | --------------------------------------- | ------------------------------------------- |
| **Private key (encrypted)** | `sops/secrets/<machine>-age.key/secret` | Generated by `age-keygen`, stored encrypted |
| **Public key**              | `sops/machines/<machine>/key.json`      | Used to encrypt secrets for this machine    |
| **Deployed private key**    | `/var/lib/sops-nix/key.txt` on machine  | Uploaded by `clan vars upload`              |

### Key Types Comparison

| Aspect                   | Clan-Generated Key (Default)                               | SSH-Derived Key (Alternative)                        |
| ------------------------ | ---------------------------------------------------------- | ---------------------------------------------------- |
| **Generation**           | `age-keygen` via clan CLI                                  | `ssh-to-age` from SSH host key                       |
| **Private key location** | Encrypted in repo, deployed to `/var/lib/sops-nix/key.txt` | `/etc/ssh/ssh_host_ed25519_key` (already on machine) |
| **When created**         | First time secrets are generated for machine               | When machine is installed                            |
| **sops-nix config**      | `sops.age.keyFile = "/var/lib/sops-nix/key.txt"`           | `sops.age.sshKeyPaths = ["/etc/ssh/..."]`            |

**Important**: Clan's `sops/machines/<machine>/key.json` should contain the **clan-generated** public key, NOT the SSH-derived key. Mixing these causes decryption failures.

## Issue 1: "Host key verification failed"

### Symptoms

```
[machine] Host key verification failed.
[machine] error: failed to start SSH connection to '<ip>'
[machine] Command 'nix-copy-closure ... --to root@<ip> ...' returned non-zero exit status 1.
```

### Root Cause

When `clan machines update` runs, it:

1. SSHs **to** the target machine
2. Runs `nixos-rebuild --target-host root@<ip>` **on** that machine
3. The target machine then runs `nix-copy-closure` back to itself

The host key verification failure happens **on the target machine**, not on your workstation. The target machine doesn't have its own IP in its `known_hosts`.

### Solution

Add the target machine's host key to its own `known_hosts`:

```bash
# From your workstation, run:
ssh root@<target-ip> "ssh-keyscan -H <target-ip> >> ~/.ssh/known_hosts"
```

### If Host Key Changed

If you see "REMOTE HOST IDENTIFICATION HAS CHANGED", the target machine's SSH host keys were regenerated (reinstall, etc.):

```bash
# 1. Fix on your workstation
ssh-keygen -R <target-ip>
ssh-keyscan -H <target-ip> >> ~/.ssh/known_hosts

# 2. Fix on the target machine itself
ssh root@<target-ip> "ssh-keygen -R <target-ip>; ssh-keyscan -H <target-ip> >> ~/.ssh/known_hosts"
```

## Issue 2: SOPS Decryption Failure at Runtime

### Symptoms

```
sops-install-secrets: Imported /etc/ssh/ssh_host_ed25519_key as age key with fingerprint age1xxx...
sops-install-secrets: failed to decrypt '...': Error getting data key: 0 successful groups required, got 0
Activation script snippet 'setupSecrets' failed (1)
```

### Root Cause

The machine's deployed age key (`/var/lib/sops-nix/key.txt`) doesn't match the key secrets are encrypted to. Common causes:

1. **Key mismatch**: `sops/machines/<machine>/key.json` has a different key than what's in `sops/secrets/<machine>-age.key/secret`
2. **SSH-derived vs generated**: `key.json` was manually set to SSH-derived key, but secrets use clan-generated key
3. **Vars not uploaded**: The age key wasn't deployed to `/var/lib/sops-nix/key.txt`

### Debugging Steps

1. **Check what key is deployed on the machine**:

   ```bash
   # Get public key from deployed private key
   ssh root@<target-ip> "nix-shell -p age --run 'age-keygen -y /var/lib/sops-nix/key.txt'"
   ```

2. **Check what key is in the repo** (`key.json`):

   ```bash
   cat sops/machines/<machine>/key.json
   ```

3. **Check what key is in the age.key secret**:

   ```bash
   # Decrypt the secret and get its public key
   nix develop -c clan secrets get <machine>-age.key | nix develop -c age-keygen -y -
   ```

4. **Check what keys a secret is encrypted to**:

   ```bash
   head -30 vars/per-machine/<machine>/openssh/ssh.id_ed25519/secret
   # Look at "recipient" fields in the "age" array
   ```

**All three should match**: deployed key = key.json = age.key secret's public key

### Solution

If there's a mismatch, you need to align all three:

**Option A: Use the clan-generated key (recommended)**

```bash
# 1. Get public key from the age.key secret
PUB_KEY=$(nix develop -c clan secrets get <machine>-age.key | nix develop -c age-keygen -y -)

# 2. Update key.json to match
echo "[{\"publickey\": \"$PUB_KEY\", \"type\": \"age\"}]" > sops/machines/<machine>/key.json

# 3. Re-encrypt all secrets with correct key
clan vars fix <machine>

# 4. Re-encrypt the age.key secret itself (add machine as recipient)
nix develop -c clan secrets get <machine>-age.key | \
  nix develop -c clan secrets set <machine>-age.key --machine <machine> --user <admin-user>

# 5. Upload and deploy
clan vars upload <machine>
clan machines update <machine>
```

**Option B: Regenerate everything from scratch**

```bash
# WARNING: This will generate new keys and invalidate existing deployments
clan vars generate <machine> --regenerate
```

## Issue 3: `clan vars upload` Fails on Target Machine

### Symptoms

Running `clan vars upload <machine>` **on the target machine itself** fails with:

```
Failed to get the data key required to decrypt the SOPS file.

Group 0: FAILED
  age1xxx...: FAILED
    - | failed to create reader for decrypting sops data key with
      | age: no identity matched any of the recipients.
```

### Root Cause

`clan vars upload` needs to decrypt `sops/secrets/<machine>-age.key/secret` to get the machine's private key. This secret is typically encrypted only for **admin users**, not for the machine itself.

**By design**: You should run `clan vars upload` from an **admin workstation** that has the decryption key (in `~/.config/sops/age/keys.txt`), not from the target machine.

### Solution A: Run from Admin Workstation (Recommended)

```bash
# From your workstation (not the target machine)
clan vars upload <machine>
```

### Solution B: Enable Machine Self-Upload

If you want the machine to upload its own secrets (for automation), add the machine as a recipient to its own age.key secret:

```bash
# 1. Ensure machine key in key.json matches age.key secret
PUB_KEY=$(nix develop -c clan secrets get <machine>-age.key | nix develop -c age-keygen -y -)
echo "[{\"publickey\": \"$PUB_KEY\", \"type\": \"age\"}]" > sops/machines/<machine>/key.json

# 2. Add machine as recipient to its own age.key secret
nix develop -c clan secrets machines add-secret <machine> <machine>-age.key

# 3. Re-encrypt the secret with the new recipient
nix develop -c clan secrets get <machine>-age.key | \
  nix develop -c clan secrets set <machine>-age.key --machine <machine> --user <admin-user>

# 4. Commit and push
git add -A && git commit -m "Allow <machine> to self-upload secrets"
git push

# 5. On the target machine, ensure the age key is in SOPS search path
# Copy the deployed key to user's sops directory:
sudo cat /var/lib/sops-nix/key.txt > ~/.config/sops/age/keys.txt
chmod 600 ~/.config/sops/age/keys.txt

# 6. Now the machine can self-upload
cd ~/dotfiles && git pull
clan vars upload <machine>
```

### Note on Security

Allowing machines to self-upload means a compromised machine could potentially extract secrets from the repository. For most use cases, running `clan vars upload` from an admin workstation is preferred.

## Issue 4: Clock Skew Warnings

### Symptoms

```
tar: key.txt: time stamp 2026-01-31 19:14:01 is 9.21 s in the future
```

### Root Cause

The target machine's clock is out of sync with your workstation. Common in WSL environments.

### Solution

On the target machine (especially WSL):

```bash
# Option 1: Sync from hardware clock
sudo hwclock -s

# Option 2: Use NTP
sudo ntpdate time.windows.com

# Option 3: For WSL specifically
wsl --shutdown  # Then restart WSL
```

## Issue 5: nftables Syntax Errors

### Symptoms

```
ruleset.conf:13:25-30: Error: syntax error, unexpected accept
  iifname "enp5s0" icmp accept
                        ^^^^^^
```

### Root Cause

Invalid nftables rule syntax. Common issues:

- `icmp accept` → needs type specifier: `icmp type { echo-request, echo-reply } accept`
- `icmpv6 accept` → needs type specifier: `icmpv6 type { ... } accept`
- `rt mss` → should be `rt mtu` for MSS clamping

### Solution

Fix the nftables rules in your router/firewall module. See `modules/router/firewall.nix` for examples.

## Issue 6: Sysctl Conflicts

### Symptoms

```
error: The option `boot.kernel.sysctl."net.ipv6.conf.all.forwarding"' is defined multiple times
```

### Root Cause

Multiple modules (e.g., router module, WireGuard, tailscale) are setting the same sysctl value.

### Solution

Use `lib.mkForce` for values that must override others, or `lib.mkDefault` for values that should be overridable:

```nix
boot.kernel.sysctl = {
  # Use mkForce when this module MUST have this value
  "net.ipv4.conf.all.forwarding" = lib.mkForce 1;

  # Use mkDefault when this is a sensible default but can be overridden
  "net.ipv6.conf.all.forwarding" = lib.mkDefault 1;
};
```

## Issue 7: `.home.arpa` Hostnames Don't Resolve on macOS

### Symptoms

```
ping pp-nas1.home.arpa
ping: cannot resolve pp-nas1.home.arpa: Unknown host
```

But the same hostname resolves fine from the router itself.

### Root Cause

macOS aggressively caches negative DNS responses (NXDOMAIN). If a `.home.arpa` lookup fails once (e.g., before the DDNS sync has populated Unbound), macOS caches the failure and continues returning "Unknown host" even after the DNS record exists.

This commonly happens after deploying router config changes — Unbound restarts and loses dynamic DDNS records until the sync re-adds them.

### Solution

Flush the macOS DNS cache:

```bash
sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder
```

If records are also missing on the router (verify with `dig @10.0.0.1 <hostname>.home.arpa`), force a DDNS re-sync:

```bash
ssh root@pp-router1.pp-wg "rm -f /run/kea-unbound-sync/records && systemctl start kea-unbound-sync"
```

### Verification

```bash
# Confirm the record exists on the router
dig @10.0.0.1 pp-nas1.home.arpa +short

# Confirm local resolution works
dig pp-nas1.home.arpa +short
ping pp-nas1.home.arpa
```

## General Debugging Tips

1. **Add `-v` for verbose SSH output**:

   ```bash
   NIX_SSHOPTS="-v" clan machines update <machine>
   ```

2. **Check the full command being run**:
   The error output shows the exact SSH and nixos-rebuild commands. Try running them manually.

3. **Test SSH connectivity separately**:

   ```bash
   ssh root@<target-ip> hostname
   ```

4. **Test nix-copy-closure separately**:

   ```bash
   nix-copy-closure --to root@<target-ip> /nix/store/some-path
   ```

5. **Check clan configuration**:
   ```bash
   grep -r "targetHost\|buildHost" machines/<machine>/
   ```

## Useful Clan Commands

### Secrets Management

```bash
# List all secrets
clan secrets list

# Get a secret's decrypted value
clan secrets get <secret-name>

# Set/re-encrypt a secret with specific recipients
clan secrets set <secret-name> --machine <m> --user <u>

# Add a machine as recipient to a secret
clan secrets machines add-secret <machine> <secret-name>

# List machines that can decrypt secrets
clan secrets machines list

# Get a machine's public key
clan secrets machines get <machine>
```

### Vars Management

```bash
# List vars for a machine
clan vars list <machine>

# Check if vars are up to date
clan vars check <machine>

# Re-encrypt vars with updated machine keys
clan vars fix <machine>

# Upload secrets to a machine
clan vars upload <machine>

# Generate/regenerate vars
clan vars generate <machine>
clan vars generate <machine> --generator <name> --regenerate
```

### Machine Management

```bash
# List machines
clan machines list

# Show machine info
clan machines show <machine>

# Update/deploy a machine
clan machines update <machine>
```
